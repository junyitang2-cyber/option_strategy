# Sector Restructure + Research Bridge Design

**Date:** 2026-06-10  
**Status:** Approved  
**Scope:** Restructure Learning Hub from Month 1-5 numbering to Sector A-E topology; merge Month 3 + Month 4 into Sector C; add new Sector D (Research Bridge) powered by financial analysis skills.

---

## Background

The D1 Learning Hub currently organises content into Month 1-5 (plus Professional Sprint as Month 6). The month numbering implies a strict chronological sequence and does not reflect the domain-based nature of the content. Additionally, five new financial skills have been installed (`/earnings`, `/sector`, `/comps`, `/ic-memo`, `/thesis-tracker`) that can generate high-quality equity research content. The goal of this redesign is to:

1. Rename the module topology from time-based months to domain-based sectors.
2. Merge the tightly coupled Vol Framework (Month 3) and Dealer/Market Making (Month 4) content into a single Sector C.
3. Introduce a new Sector D (Research Bridge) that bridges equity research outputs to options strategy decisions, with content generated by the financial skills.

---

## Sector Mapping

| Sector | Name | Previous | Content |
|---|---|---|---|
| A | Risk Mechanics | Month 1 | Greeks, delta/gamma/vega/theta/rho, sensitivity, decay |
| B | Trade Construction | Month 2 | Strategy construction, spreads, straddles, condors, butterflies, client recommendation drills |
| C | Market Dynamics | Month 3 + Month 4 | Vol Framework, RV/IV, event vol, equity skew, term structure, vol surface; Dealer workflow, inventory, delta hedging, gamma scalping, vega buckets, P&L attribution |
| D | Research Bridge | NEW | Earnings analysis, sector analysis, comps, IC memos, thesis tracking → options expression |
| E | Complex Products | Month 5 | Exotics, structuring, Asian/barrier/quanto/digital/autocallable, structured product workflow |
| — | Professional Sprint | Month 6 (unchanged) | Topic-filtered sprint questions, self-scoring, weak-topic tracking — kept as standalone feature, no sector letter |

**Rationale for C merger:** Vol Framework and Dealer/Market Making are conceptually inseparable in practice — dealers use vol models to price and hedge, and vol judgements directly drive dealer behaviour. Separating them into two months was an implementation artefact, not a pedagogical choice.

**Rationale for D as new:** The equity research skills produce structured outputs (earnings previews, sector reports, comps tables, IC memos, investment theses) that are not currently represented in the platform. Sector D creates a dedicated home for the "research view → options expression" learning chain.

## Architecture

Three independent layers of change, each implementable in sequence:

```
Layer 1: Data Layer (data/learning-content.js)
  → Rename month field to sector
  → Remap numeric values to letters (1→A, 2→B, 3→C, 4→C, 5→E)
  → No structural changes to content objects

Layer 2: UI Layer (app.js + index.html)
  → Update formatLearningPeriod, LEARNING_UI_TEXT, filter state
  → Register new research-bridge tab
  → Update roadmap rendering for Sector A-E

Layer 3: Sector D Content (data/research-bridge-content.js + app.js render)
  → New data file with researchCases and viewToTradeDrills
  → New renderLearningResearchBridge() function
  → New panel in index.html
```

## Layer 1: Data Layer

### `data/learning-content.js`

All module, scenario, and client drill objects currently carry a `month` field with numeric values 1–6. This field is renamed to `sector` and the values are remapped:

| Old value | New value | Reason |
|---|---|---|
| `month: 1` | `sector: "A"` | Sector A |
| `month: 2` | `sector: "B"` | Sector B |
| `month: 3` | `sector: "C"` | Sector C (Vol half) |
| `month: 4` | `sector: "C"` | Sector C (Dealer half) |
| `month: 5` | `sector: "E"` | Sector E |
| `month: 6` | `sector: "sprint"` | Professional Sprint module object — field renamed for consistency; sprint question objects in `professionalSprintQuestions[]` do not carry a sector field and are unchanged |

The change is a mechanical global find-and-replace with value remapping. No content is added, removed, or reordered.

### `data/research-bridge-content.js` (new file)

New file exporting two arrays: `researchCases` and `viewToTradeDrills`.

**`researchCases` schema:**

```js
{
  id: String,           // e.g. "rc-earnings-nvda-001"
  skill: String,        // "earnings" | "sector-analysis" | "comps" | "ic-memo" | "thesis"
  sector: "D",
  ticker: String,       // e.g. "NVDA"
  title: String,        // English title
  titleCn: String,      // Chinese title
  context: String,      // Research summary (2-4 sentences)
  contextCn: String,
  keyView: String,      // Core directional / vol view
  keyViewCn: String,
  impliedMove: String,  // e.g. "±8.2%" — for earnings cases; null for others
  suggestedStructures: [String],  // e.g. ["short iron condor", "long straddle"]
  rationale: String,
  rationaleCn: String,
}
```

**`viewToTradeDrills` schema:**

Reuses the existing client drill step-reveal pattern:

```js
{
  id: String,           // e.g. "vtd-001"
  sector: "D",
  researchInput: String,   // The research context presented to learner
  researchInputCn: String,
  question: String,        // The options structure question
  questionCn: String,
  steps: [               // Step-by-step reveal, same model as clientDrills
    { key: "view",          label: String, labelCn: String, content: String, contentCn: String },
    { key: "constraints",   label: String, labelCn: String, content: String, contentCn: String },
    { key: "candidates",    label: String, labelCn: String, content: String, contentCn: String },
    { key: "recommendation",label: String, labelCn: String, content: String, contentCn: String },
    { key: "risks",         label: String, labelCn: String, content: String, contentCn: String },
    { key: "expression",    label: String, labelCn: String, content: String, contentCn: String },
  ],
}
```

**Initial content targets:**

| `/sector` (Sector Analysis) | 3 | 3 |
| `/comps` | 3 | 2 |

Plus 20-30 new Sector D scenarios added to `learning-content.js` Scenario Bank (tagged `sector: "D"`), generated from skill outputs.

**Sub-section 3: (none — Sector D scenarios surface via main Scenarios tab with sector filter)**

No duplicate scenario rendering inside Sector D tab. The Scenario Bank's sector filter (value "D") is the canonical entry point.

## Skills-as-Content-Generation Workflow

Financial skills are used offline to produce content, not called at runtime:

```
1. Run /earnings <TICKER>       → earnings preview report
   Extract: implied move, key view, risk factors
   Format as researchCase + viewToTradeDrill in research-bridge-content.js

2. Run /sector <SECTOR>         → sector analysis report
   Extract: sector dynamics, relative valuation, options implications
   Format as researchCase + viewToTradeDrill

3. Run /comps <TICKER>          → comparable company analysis
   Extract: relative value view, mispricing hypothesis
   Format as researchCase + viewToTradeDrill

4. Run /ic-memo <TICKER>        → initiating coverage note
   Extract: 12-month thesis, PT, key risks
   Format as researchCase + viewToTradeDrill

5. Run /thesis-tracker          → investment thesis summary
   Extract: current view, conviction level, time horizon
   Format as researchCase + viewToTradeDrill
```

Each skill run produces one batch of content. The platform does not call these skills at runtime — it only renders the pre-formatted content objects.
